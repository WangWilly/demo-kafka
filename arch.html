架構師的 Message Broker 選型字典

從底層哲學到實戰避坑，一頁掌握 RabbitMQ 與 Apache Kafka 的核心設計原則，不再為選型與併發問題發愁。

核心決策矩陣 (Choose Your Fighter)

RabbitMQ

Smart Broker / Dumb Client

  • 模型： 推送 (Push) & 消費後刪除。
  • 強項： 複雜的路由規則 (Exchange)、任務佇列、精確控制單筆訊息。
  • 弱項： 訊息堆積會導致效能崩盤、歷史重播困難。
最佳場景

微服務非同步解耦、寄發 Email/SMS、耗時背景任務排隊、訂單狀態機。

Apache Kafka

Dumb Broker / Smart Client

  • 模型： 拉取 (Pull) & 不可變日誌保留。
  • 強項： 海量吞吐量、嚴格保序、無限廣播、歷史資料重播。
  • 弱項： 架構較重、無法精確單筆刪除、擴容 Partition 易洗牌。
最佳場景

事件溯源 (Event Sourcing)、網站活動追蹤、即時大數據串流分析、Log 收集。

RabbitMQ 開發 4 大避坑指南

1

保護 Consumer：設定 QoS

預設情況下 Broker 會把訊息全部狂塞給 Consumer 導致 OOM。必須設定背壓機制。

channel.basicQos(prefetch_count=20)
2

絕不掉單：三重持久化保證

追求可靠性時，不可依賴預設的記憶體模式，必須在三個節點都上鎖。

  • Producer: Publisher Confirms
  • Broker: durable=true & delivery_mode=2
  • Consumer: 關閉 autoAck，手動 basic.ack
3

防禦毒藥：死信交換機 (DLX)

遇到會讓程式崩潰的錯誤資料，切勿無限 Requeue 導致系統卡死 (Head-of-line blocking)。

basic.reject(requeue=false) -> DLX -> DLQ
4

高可用架構：放棄鏡像，改用 Quorum

傳統的鏡像佇列 (Mirrored Queues) 容易引發腦裂。叢集部署請一律採用基於 Raft 演算法的 Quorum Queues (仲裁佇列)。

Kafka 設計 4 大避坑指南

1

Partition 數量：寧多勿少，但別破萬

同一個 Group 內，Consumer 數量不能大於 Partition 數。擴容 Partition 會導致 Hash 跑版，因此初期應適度超配 (如 32/64 個)。

2

Key 的設計：提防資料傾斜 (Data Skew)

為求嚴格保序使用 `UserID` 作為 Partition Key 時，要當心「超級大戶」引發熱點問題，導致單一 Partition/Consumer 被灌爆。

3

遠離死亡螺旋：避免 Rebalance 風暴

Consumer 處理太久會被踢出 Group 觸發全局暫停 (Stop-the-world)。

解法：將超耗時任務丟給內部 Thread Pool，讓 Consumer 主執行緒盡快去 poll() 下一批。

4

防禦重複消費：關閉自動提交

絕對禁止在嚴謹業務中使用自動提交 Offset。必須在 DB Transaction 寫入成功後才手動提交。

enable.auto.commit=false -> commitSync()

進階架構：冪等性 (Idempotency) 與高併發

為什麼需要冪等性？

在分散式系統中，網路是不可靠的。Broker 只能保證「至少傳遞一次 (At-least-once)」。當 Consumer 處理完畢但回傳 ACK 失敗時，Broker 會重新派發，導致重複消費（例如重複扣款）。

方案 A: Unique Key (DB)

最簡單穩定的做法。利用資料庫的唯一約束來阻擋重複資料。

INSERT INTO processed_events (msg_id)
-- 若違反 Unique 則 Catch Exception 並忽略

方案 B: 原子鎖 (Redis SETNX)

解決高併發下的 Race Condition。當兩個 Consumer 在同一毫秒收到相同訊息時，傳統的 Check-then-Act 會失效。

if (redis.setnx(msg_id, 1)) {
  // 獲得鎖，執行業務
} else {
  // 攔截！重複或併發
}

方案 C: User Affinity (物理防禦)

從物理上消滅併發。將同一 User 的訊息永遠路由到同一個 Queue/Partition，由單一執行緒單線程處理。

  • Kafka: 原生 Partitioner (Hash by Key)
  • RabbitMQ: 安裝 consistent-hash-exchange 外掛

終極大絕招：序號控制 (Sequence ID / 樂觀鎖)

若要同時解決「重複消費」與「順序錯亂 (Out-of-Order)」，在 DB 維護一個 expected_seq。
只有當 Msg.Seq == DB.ExpectedSeq 時才執行。未來訊息 (大於期望值) 退回重試，過去訊息 (小於期望值) 視為重複直接忽略。
